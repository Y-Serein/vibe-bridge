本文件是 `vibe-bridge` 的项目级规则，不是全局规则。全局规则只放在 `/home/rv_nano/.codex/AGENTS.md`；如果冲突，优先遵守本文件中更具体的项目约束。

## 0. 工程控制论式项目执行协议

本项目任务默认按闭环推进：目标 -> 状态 -> 误差 -> 控制动作 -> 反馈 -> 修正 -> 验证 -> 沉淀。

- 动手前先说明当前目标、系统状态、目标与现状误差、本轮控制变量和最小闭环验证方式。
- 只调整完成目标所需的最小变量，不把无关优化、重构或格式化混进当前任务。
- 修改后必须用单测、Windows doctor、daemon 运行反馈、板端反馈或用户实测验证；反馈不一致时先定位误差来源，再继续改。
- 不破坏已跑通的 wrapper-PTY、session grid、daemon、board-assigned session id、HID/board 协议、终端 replay 和 agent discovery 路径。
- 长任务或上下文可能丢失时，按项目规则更新 HANDOFF，而不是依赖对话记忆。

辅助工具：

- 开工前优先运行只读 preflight：`python3 /home/rv_nano/Sipeed/T_tools/agent_preflight.py --project vibe-bridge`。
- host/daemon/session/HID/board 协议任务优先读取 `/home/rv_nano/Sipeed/.claude/skills/rv-nano-vibe-tools/SKILL.md`。
- 遇到重复问题先查 `/home/rv_nano/Sipeed/C_context/KNOWN_FAILURES.md`。

## 14.1 跨端状态机和闭环复核

`vibe-bridge` 不是单独的 Python 工具，它和板端 `aikb_hid_input`、`aikb_lcd_ui`、
HID gadget、FIFO、Session UI、VT100 replay 组成一条产品闭环。涉及以下内容时，
不能只看当前文件或当前函数就改：

- wrapper / bootstrap / daemon 启停逻辑。
- `SESSION` 面板、active sid、VT100 replay、window switch。
- encoder / 按键路由。
- real-hidraw、mock daemon、stale socket/state 的 fallback。
- host dashboard JSON 和 board terminal/pet/dashboard view 的切换。

动手前必须先用 3-6 条写出当前闭环：

- 输入事件是什么，来自 board 还是 wrapper。
- host 哪一层拥有状态，board 哪一层负责显示。
- 数据最终写到哪里：HID、FIFO、dashboard JSON、VT100 stream 或 pet event。
- 进入、确认、取消、退出分别由谁处理。
- 哪些旧 fallback 或 mock 路径可能抢控制权。

改完后必须做架构一致性复核：

- 复查相邻模块，不只看被改函数。
- 明确单测覆盖的是哪一段闭环，未覆盖的真实 HID / 板端显示风险要写出来。
- 如果改了 session 或 UI 所有权，必须验证普通 `codex` 启动不会卡死、mock 不会污染 real-hidraw、VOICE/encoder 不会被误路由。
- 如果现场出现“多 daemon / 状态脏 / 多 UI 抢屏 / 延迟很大 / 按键要按多次”这类系统性现象，先停止局部补丁，回到全链路状态图和日志证据。

## 16. 优先级规则

当规则冲突时，按以下优先级执行：

1. 安全和隐私
2. 不破坏我已有的改动
3. 不违反 CDTR 目录组织
4. 最小可回滚改动
5. 正确性和稳定性
6. 与现有代码风格一致
7. 可验证性
8. 简洁表达
9. 美观、优化或重构

特别说明：

- 如果 “快速完成” 和 “安全验证” 冲突，以安全验证为准。
- 如果 “主动修复” 和 “任务边界” 冲突，以任务边界为准。

## 17. 总体要求

你不是来“展示能力”的，而是来安全、准确、可控地完成任务。

不要擅自扩大任务范围。
不要过度设计。
不要编造。
不要隐藏风险。
不要为了让结果看起来完成而跳过验证。
不要破坏我已有的工作。
不要把临时文件堆在项目根目录。
不要把概念性问题答完就丢，应该沉淀到 `C_context/QA_Log.md`。
